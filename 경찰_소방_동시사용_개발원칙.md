# 경찰_소방_동시사용_개발원칙

> 최우선 기준: `D:\hankuk\AGENTS.md`
> 목적: 경찰/소방/지점 등을 동시에 운영하는 구조를 어떻게 설계할지 한 번에 이해하기 위한 보조 문서

---

## 1. 기본 원칙

앞으로의 기준은 아래 두 줄입니다.

- 서비스는 앱으로 나눈다.
- 같은 서비스 안의 경찰/소방/지점은 앱 안의 runtime tenant로 처리한다.

즉:

- `study-hall`, `score-predict`, `interview-pass`는 서로 다른 앱
- 그 앱 안의 `police`, `fire`, `gangnam` 같은 것은 tenant/division/workspace

---

## 2. 왜 이렇게 가나

이 구조가 좋은 이유:

- 서비스별로 배포와 운영을 분리할 수 있다.
- 같은 서비스 안의 지점/브랜드/권한 차이를 앱 복제 없이 처리할 수 있다.
- 로그인과 공통 사용자 정보는 공유하기 쉽다.
- tenant가 늘어나도 Vercel 프로젝트와 Supabase 프로젝트가 같이 폭증하지 않는다.

---

## 3. 기준 패턴

`study-hall`의 `[division]` 방식이 기준 패턴입니다.

즉 tenant는:

- path slug
- request 문맥
- auth/session 문맥
- DB 필터

로 처리합니다.

반대로 `NEXT_PUBLIC_TENANT_TYPE`처럼 build 시점에 경찰/소방을 고정하는 방식은 과도기 방식으로만 봅니다.

---

## 4. 런타임 tenant 처리 원칙

권장 순서:

1. path slug
2. runtime hostname
3. request 기반 lookup

예:

- `/police/...`
- `/fire/...`
- `/gangnam/...`

핵심은 하나의 앱 인스턴스가 여러 tenant를 동시에 처리할 수 있어야 한다는 점입니다.

---

## 5. DB 원칙

- Supabase 프로젝트는 하나
- 공통 인증과 사용자 정보는 `public`
- 앱별 데이터는 앱별 schema
- 같은 앱 안 tenant 차이는 schema를 더 늘리는 것이 아니라 row/config/filter로 분리

예:

- `study_hall` schema 안에서 `division_id`로 구분
- `interview` schema 안에서 `tenant_slug`로 구분

---

## 6. 앱별 해석

### `study-hall`

- 이미 runtime division 구조에 가깝다
- 앞으로의 기준점으로 삼는다

### `score-predict`

- 현재 build-time tenant 흔적이 있다
- 장기적으로는 runtime tenant 구조로 옮겨야 한다
- 다만 계산 규칙 차이가 너무 크면 내부 도메인 모듈 분리가 더 중요할 수 있다

### `interview-pass`

- 현재 build-time tenant 흔적이 있다
- 라벨/브랜딩/필드 차이를 config 기반 runtime tenant로 옮기는 것이 바람직하다

---

## 7. 절대 기준

- 경찰/소방 차이만으로 새 앱을 만들지 않는다
- 경찰/소방 차이만으로 새 Vercel 프로젝트를 만들지 않는다
- 경찰/소방 차이만으로 새 Supabase 프로젝트를 만들지 않는다
- 같은 서비스라면 기존 앱 안 tenant 확장을 먼저 본다

---

## 8. 한 줄 결론

Hankuk의 최적 구조는:

- 앱은 서비스별로 분리
- tenant는 앱 내부에서 runtime 처리
- 인증과 Supabase 프로젝트는 공유
- 앱별 데이터는 schema 분리

입니다.
