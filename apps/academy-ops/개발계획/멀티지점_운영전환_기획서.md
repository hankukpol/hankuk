# 멀티지점 운영 전환 기획서
## 한국경찰학원 통합 관리 시스템 → 복수 지점 독립 운영 구조로 전환

**작성일**: 2026-03-21
**버전**: v1.0
**목적**: 하나의 코드베이스에서 경찰·소방 등 복수 학원 지점을 독립 운영하는 구조 설계

---

## 핵심 운영 원칙 (변경 불가)

> **원칙 1 — 코드 단일화**
> 코드 수정은 반드시 모든 지점에 동일하게 반영된다.
> 지점별로 코드를 따로 관리하지 않는다.

> **원칙 2 — 설정 독립화**
> 모든 기능의 활성화/비활성화, 세부 설정값은 각 지점 관리자가 독립적으로 설정한다.
> 타 지점 설정에 영향을 주지 않는다.

> **원칙 3 — 데이터 격리**
> 각 지점의 학생, 성적, 결제, 출결 데이터는 완전히 분리된다.
> 슈퍼관리자만 전 지점 데이터를 조회할 수 있다.

## 1. 현재 구조와 목표 구조 비교

### 현재 (단일 학원)
```
[한국경찰학원] ────── 관리자 ── 학생/결제/성적 (단일 DB)
                └─── 설정  ── 학원정보 싱글톤 (id=1 고정)
```

### 목표 (다중 지점)
```
[슈퍼관리자] ─────── 전 지점 조회·관리
     │
     ├── [경찰학원] ── 지점관리자 ── 학생/결제/성적 (격리된 데이터)
     │               └── 지점설정  ── 독립 설정값
     │
     ├── [소방학원] ── 지점관리자 ── 학생/결제/성적 (격리된 데이터)
     │               └── 지점설정  ── 독립 설정값
     │
     └── [신규지점] ── (슈퍼관리자가 생성)
```

## 2. 관리자 권한 구조

### 권한 계층 (변경 없음)
기존 AdminRole 8단계 유지:
```
VIEWER(0) < TEACHER(1) < COUNSELOR(2) < ACADEMIC_ADMIN(3)
< MANAGER(4) < DEPUTY_DIRECTOR(5) < DIRECTOR(6) < SUPER_ADMIN(7)
```

### 권한별 지점 접근 범위

| 역할 | 접근 범위 | 설명 |
|------|-----------|------|
| **SUPER_ADMIN** | **전 지점** | 모든 지점 데이터 조회 + 지점 생성/삭제 + 지점 전환 |
| DIRECTOR (원장) | 소속 지점만 | 소속 지점 전체 접근 + 지점 설정 변경 |
| DEPUTY_DIRECTOR ~ VIEWER | 소속 지점만 | 각 역할에 부여된 기능 범위 내에서 소속 지점만 접근 |

### 지점 전환 UI (SUPER_ADMIN 전용)
- 화면 상단 헤더에 `[지점명 ▼]` 드롭다운 표시
- 다른 관리자는 소속 지점명만 텍스트로 표시
- "전체 보기" 모드: 모든 지점 통합 집계 조회 가능

## 3. 데이터베이스 변경 설계

### 3-1. 신규 테이블: `academies` (지점 마스터)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | INT PK | 지점 고유번호 |
| code | STRING UNIQUE | 식별 코드 (예: `gyeongchal`, `sobang`) |
| name | STRING | 지점명 (예: 한국경찰학원) |
| type | ENUM | POLICE / FIRE / CIVIL_SERVICE / OTHER |
| isActive | BOOLEAN | 활성화 여부 |
| createdAt | DATETIME | 생성일시 |

### 3-2. 신규 테이블: `academy_settings` (지점별 독립 설정)

> **기존 `academy_settings` 싱글톤 테이블을 지점별로 확장**

| 컬럼 그룹 | 포함 항목 |
|-----------|-----------|
| **학원 기본정보** | 학원명, 원장명, 주소, 전화, 팩스, 홈페이지, 사업자번호, 학원등록번호, 계좌정보, 직인/로고 이미지 |

| **승인 임계값** | 환불 승인 기준금액, 할인 승인 기준금액, 현금 승인 기준금액 |

| **성적 처리 규칙** | JSON - 합격 기준점수, 우수 기준점수, 순위 표시 여부 |

|------|------|------|

### 3-5. DB 마이그레이션 실행 순서

```
Step 1: academies 테이블 생성 + 한국경찰학원 1행 삽입 (id=1)
Step 2: academy_settings 테이블 구조 변경 (싱글톤 → 지점별)
Step 3: 핵심 테이블에 academyId 컬럼 추가 (DEFAULT 1, NOT NULL)
Step 4: 나머지 테이블에 academyId 컬럼 추가
Step 5: exam_subjects 테이블 생성 + 경찰학원 기본 과목 삽입
Step 6: admin_users.academyId 설정 (SUPER_ADMIN은 NULL)
```

## 4. 인증·세션 변경 설계

### 4-1. 로그인 처리 흐름 변경

```
[현재]
로그인 → AdminUser 조회 → role 확인 → 접근 허용/거부

[변경 후]
로그인 → AdminUser 조회 → academyId + role 확인
        → academyId를 세션 컨텍스트에 저장
        → 이후 모든 DB 쿼리에 academyId 자동 필터 적용
```

### 4-2. 세션 컨텍스트 구조 (개념)

```
AdminContext {
  userId         // Supabase Auth ID
  adminUser      // DB 관리자 레코드
  academyId      // 소속 지점 ID (SUPER_ADMIN이면 null)
  isSuperAdmin   // 슈퍼관리자 여부
  activeAcademy  // 현재 선택된 지점 (SUPER_ADMIN이 전환 시 변경)
}
```

### 4-3. SUPER_ADMIN 지점 전환

- **쿠키**에 `activeAcademyId` 저장
- SUPER_ADMIN이 헤더 드롭다운에서 지점 선택 시 쿠키 업데이트
- `activeAcademyId = null` → 전체 조회 모드
- API 엔드포인트: `POST /api/admin/switch-academy`

## 7. 신규 지점 생성 프로세스

```
① 슈퍼관리자 → /admin/super/academies/new
   - 지점 코드, 학원명, 유형 입력
   - 신규 Academy 레코드 생성

② 기본 설정 초기화
   - AcademySetting 기본값으로 생성
   - 시험 과목: 지점 유형에 맞는 기본 과목 자동 삽입
     (경찰: 헌법·형사법·경찰학 / 소방: 소방학개론·소방관계법규)

③ 지점 관리자 계정 생성
   - AdminUser 생성 + academyId 연결

④ 지점 관리자가 직접 세부 설정 완료
   - /admin/settings/ 하위 항목들을 지점에 맞게 입력
```
