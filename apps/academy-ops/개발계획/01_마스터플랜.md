# 통합 학원 관리 시스템 — 마스터 개발 계획서

**작성일**: 2026-03-13
**버전**: v1.4 (2026-03-16 인터뷰 2라운드 반영)
**오픈 목표**: 2026년 6월 이전 (수강 등록·수납 관리 우선)
**프로젝트명**: 아침모의고사 → 공무원학원 통합 관리 시스템 (가칭: AcadPro)
**기반 기술**: Next.js 14 App Router, TypeScript, Prisma 6, PostgreSQL (Supabase), TailwindCSS

> 이 문서는 현재 운영 중인 **아침모의고사 성적·출결관리 시스템**을 기반으로, 공무원 학원 전용 **통합 관리 플랫폼**으로 확장하기 위한 전체 개발 계획이다.
> 각 모듈은 독립적으로 개발하고 단계적으로 통합하는 방식으로 진행한다.

## 1. 현재 시스템 자산 및 재활용 계획

### 1-1. 재활용 가능한 기존 기능

| 기존 기능 | 재활용 범위 |
|---|---|
| 학생(수험번호) 관리 | 수강생 마스터로 확장 |
| 면담 기록 | 수강 상담 + 재학생 면담 통합 |
| 포인트 시스템 (개근/성적) | 수강 포인트 시스템으로 확장 |
| SMS/알림 발송 | 수납 알림, 상담 알림 등으로 확장 |
| 관리자 권한 (VIEWER/TEACHER/SUPER_ADMIN) | 직원 권한 체계로 확장 |
| 감사 로그 | 직원별 작업 이력으로 확장 |
| 아침모의고사 성적·출결 시스템 | 성적 관리 모듈 중 한 축 |

### 1-2. 기존 시스템과의 통합 방식

```
기존 시스템 (아침모의고사)          신규 시스템 (통합 관리)
───────────────────────           ──────────────────────────────
Student (수험번호 기반)  ◀── 연동 ──▶  AcademyMember (회원 마스터)
CounselingRecord        ◀── 통합 ──▶  ConsultationRecord
PointLog (개근/성적)    ◀── 통합 ──▶  PointLedger (포인트 원장)
AdminUser (관리자)      ◀── 확장 ──▶  Staff (직원 + 권한)
```

**핵심 원칙**: 기존 `Student` 테이블은 유지하고, 새로운 `AcademyMember` 테이블을 만들어 `examNumber`로 연결하는 방식을 사용한다. 기존 시스템을 깨지 않고 점진적으로 확장한다.

## 3. 모듈 1: 수강관리

### 3-1. 기능 개요

방문 상담부터 수강 등록, 재원, 휴원, 퇴원까지 학원생의 전체 생애주기를 관리한다.

### 3-2. 세부 기능 목록

#### A. 회원 마스터 관리

| 기능 | 설명 |
|---|---|
| 회원 등록 | 이름, 연락처, 생년월일, 수험번호, 수강 유형, 등록 경로(인터넷/지인/간판 등) |
| 회원 검색 | 이름/연락처/수험번호로 통합 검색, 수강 상태 필터 |
| 회원 상세 조회 | 수강 이력, 수납 이력, 상담 기록, 포인트, 성적 한 페이지에서 조회 |
| 사진 등록 | 회원증 사진 또는 신분증 사진 업로드 |
| 수강 상태 | 재원 / 휴원 / 퇴원 / 졸업 상태 관리 |

#### B. 수강 상담 관리 (신규 방문자)

> **현재 면담 기록과 별도**: 기존 `CounselingRecord`는 재학생 학습 면담용. 신규는 **입원 상담(영업 상담)** 용.

| 기능 | 설명 |
|---|---|
| 상담 등록 | 방문/전화/온라인 상담 구분, 관심 강좌, 상담 내용, 담당 직원 |
| 상담 단계 | 잠재 → 상담 완료 → 등록 → 미등록(사유) |
| 팔로업 알림 | 미등록 시 X일 후 재연락 일정 등록, 알림 발생 |
| 상담 통계 | 월별 상담 건수, 등록 전환율, 담당자별 실적 |

#### C. 수강 등록/변경/퇴원

| 기능 | 설명 |
|---|---|
| 수강 등록 | 강좌 선택, 수강료 책정, 수납 방법 선택 |
| 수강 변경 | 강좌 추가/삭제, 반 변경 |
| 휴원 처리 | 휴원 기간 설정, 복귀 예정일, 사유 |
| 퇴원 처리 | 퇴원 사유 선택(자퇴/이사/합격 등), 환불 처리 연계 |
| 반 편성 | 강좌별 반 배정, 정원 관리 |

#### D. 학생 면담 통합 뷰

- 기존 `CounselingRecord` 재학생 면담 + 신규 입원 상담을 **한 타임라인에서 통합 조회**
- 학생 상세 페이지에서 전체 상담 이력 시계열로 표시

### 3-3. 주요 화면

```
/admin/members                  ← 전체 회원 목록
/admin/members/[id]             ← 회원 상세 (수강/수납/상담/성적 통합)
/admin/members/new              ← 신규 회원 등록
/admin/consultations            ← 신규 방문 상담 목록
/admin/consultations/[id]       ← 상담 상세
/admin/enrollments              ← 수강 등록/변경/퇴원 처리
```

#### B. POS 결제 화면

| 결제 취소 | 당일 결제 취소 (PG 연동 시 자동 취소) |
| 이력 조회 | 일별/직원별/강좌별 POS 거래 내역 |

## 7. 모듈 5: 포인트 시스템

### 7-1. 기능 개요

기존 개근·성적우수 포인트 시스템을 확장하여 수강료 할인, 시설 이용 등 실제 사용 가치가 있는 통합 포인트 제도로 운영한다.

### 7-2. 포인트 적립 규칙

| 적립 이벤트 | 적립량 | 비고 |
|---|---|---|
| 개근 (주간) | 기존 시스템 연동 | 아침모의고사 출결 |
| 성적우수 | 기존 시스템 연동 | 아침모의고사 성적 |

| 합격 축하 | 10,000P | 합격 보고 시 |
| 수기 지급 | 직원이 직접 지급 | 이유 기록 필수 |

### 7-3. 포인트 사용처

| 사용처 | 내용 |
|---|---|
| 수강료 차감 | 포인트 → 현금처럼 수강료에서 차감 (1P = 1원) |
| 단과 강좌 할인 | POS 결제 시 포인트 적용 |
| 스터디룸 무료 이용 | X포인트 = Y시간 무료 이용 |
| 교재 구매 | 교재 판매 시 포인트 사용 |

### 7-4. 포인트 관리 기능

| 기능 | 설명 |
|---|---|

| 유효기간 | 포인트 유효기간 설정 (기본: 1년) |

## 9. 모듈 7: 직원·권한 관리

### 9-1. 기능 개요

각 직원이 고유 계정으로 로그인하고, 수행한 모든 작업(수납, 상담, 성적 입력 등)을 직원별로 추적한다.

### 9-2. 직원 역할 정의

| 역할 | 영문 | 권한 |
|---|---|---|
| 원장 | DIRECTOR | 전체 + 정산·통계·직원 관리 |
| 교무 | ACADEMIC | 성적·출결·상담 CRUD |
| 수납 | CASHIER | 수납·결제·환불 처리 |
| 강사 | TEACHER | 담당 강좌 성적·출결만 조회·수정 |
| 조회 전용 | VIEWER | 읽기만 가능 |

> **기존 `AdminRole` 확장**: 기존 VIEWER/TEACHER/SUPER_ADMIN은 새 역할 체계로 마이그레이션. SUPER_ADMIN = DIRECTOR.

### 9-3. 권한 매트릭스

| 기능 | DIRECTOR | ACADEMIC | CASHIER | TEACHER | VIEWER |
|---|:---:|:---:|:---:|:---:|:---:|
| 회원 등록·수정 | ✓ | ✓ | ✓(기본정보) | ✗ | ✗ |
| 수납 처리 | ✓ | ✗ | ✓ | ✗ | ✗ |
| 환불 승인 | ✓ | ✗ | 신청만 | ✗ | ✗ |
| 성적 입력·수정 | ✓ | ✓ | ✗ | ✓(담당) | ✗ |
| 상담 기록 | ✓ | ✓ | ✗ | ✓ | ✗ |
| 사물함 배정 | ✓ | ✓ | ✓ | ✗ | ✗ |
| 직원 관리 | ✓ | ✗ | ✗ | ✗ | ✗ |
| 전체 매출 통계 | ✓ | ✗ | 당일만 | ✗ | ✗ |

### 9-4. 작업 이력 추적

기존 `AuditLog` 시스템을 확장하여 **누가, 언제, 무엇을 했는지** 모든 주요 작업을 기록.

| 추적 대상 | 예시 로그 |
|---|---|
| 수납 처리 | "김교무 → 홍길동 수강료 300,000원 카드 수납" |

### 13-2. 결제 연동 옵션

| 옵션 | 장점 | 단점 |
|---|---|---|
| **토스페이먼츠** | 국내 점유율 1위, 문서 좋음, 학원 서비스 사례 많음 | 월 이용료 없음, 건당 수수료 |
| **KG이니시스** | 대형 학원 다수 사용 | 연동 복잡 |
| **수동 확인** (1단계) | 개발 없이 즉시 사용 | 카드 자동화 불가 |

**권장**: 1단계는 현금/이체 수동 확인 + 카드는 별도 단말기 후 수동 입력으로 시작 → Phase 2에서 PG 연동

### 13-3. 성능 및 캐싱

- 대시보드 KPI: `unstable_cache` 60초 캐시 (기존 방식 유지)
- 수납 현황: `force-dynamic` (실시간 필요)
- 강좌 카탈로그: 1시간 캐시

### 13-4. 데이터 마이그레이션

기존 `PointLog` → `PointLedger` 마이그레이션 스크립트 필요:
- `ATTENDANCE_BONUS` → `PointSource.ATTENDANCE`
- `SCORE_BONUS` → `PointSource.SCORE`
- `MANUAL` → `PointSource.MANUAL`

### 13-5. 역할 마이그레이션

기존 `AdminRole`:
- `SUPER_ADMIN` → `DIRECTOR`
- `TEACHER` → `ACADEMIC` (강사 역할은 신규 `TEACHER` 분리)
- `VIEWER` → `VIEWER`

| 성적 데이터 | 기존 DB | 기존 시스템 존재 | 🔴 최우선 (즉시 사용 가능) |
| 출결 데이터 | 기존 DB | 기존 시스템 존재 | 🟡 권장 |

## 15. 멀티지점 운영 아키텍처 (2026-03-21 추가)

> **상세 설계**: `멀티지점_운영전환_기획서.md` 참조 (루트 폴더)
> 이 섹션은 마스터플랜 관점의 핵심 변경사항만 요약한다.

### 15-1. 아키텍처 전환 방향

```
[기존 — 단일 학원]
한국경찰학원 ─── 관리자 ─── 학생/결제/성적 (단일 DB)
                └─── 설정  ─── 학원정보 싱글톤 (id=1 고정)

[변경 후 — 멀티지점]
SUPER_ADMIN ────── 전 지점 조회·관리
     │
     ├── [경찰학원] ─── 지점관리자 ─── 학생/결제/성적 (격리된 데이터)
     │                └─── 지점설정  ─── 독립 설정값
     │
     ├── [소방학원] ─── 지점관리자 ─── 학생/결제/성적 (격리된 데이터)
     │                └─── 지점설정  ─── 독립 설정값
     │
     └── [신규지점] ─── (SUPER_ADMIN이 생성)
```

### 15-2. 전략: 애플리케이션 레벨 Row-based 멀티테넌시

| 방식 | 선택 이유 |
|------|-----------|
| **Row-based (선택)** | Prisma 현행 구조 유지, 소규모(2~5지점)에 최적, 빠른 구현 |
| Schema-per-tenant | Prisma 지원 복잡, 마이그레이션 어려움 |
| DB per 지점 | 슈퍼관리자 통합 조회 불가, 운영비 과다 |

### 15-3. DB 신규 모델

```prisma
// 지점 마스터
model Academy {
  id        Int         @id @default(autoincrement())
  code      String      @unique   // 'gyeongchal', 'sobang'
  name      String                // '한국경찰학원'
  type      AcademyType           // POLICE | FIRE | CIVIL_SERVICE | OTHER
  isActive  Boolean     @default(true)
  settings  AcademySetting?
  adminUsers AdminUser[]
  createdAt DateTime    @default(now())
}

enum AcademyType { POLICE FIRE CIVIL_SERVICE OTHER }

// 지점별 시험 과목 (Subject enum 하드코딩 대체)
model ExamSubject {
  id           Int     @id @default(autoincrement())
  academyId    Int
  code         String  // 'CONSTITUTIONAL_LAW'
  name         String  // '헌법'
  shortName    String? // '헌'
  displayOrder Int
  isActive     Boolean @default(true)
  maxScore     Int     @default(100)
}
```
