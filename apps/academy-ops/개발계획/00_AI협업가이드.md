# AI 협업 개발 가이드 (AI Team Development Guide)

**버전**: v1.0
**작성일**: 2026-03-13
**목적**: Claude Code, Claude.ai, Cursor, Copilot 등 여러 AI 에이전트가 동일한 프로젝트에서 충돌 없이 협업하는 방법을 정의한다.

---

## 1. 역할 분담

### 1-1. Claude Code (CLI / VSCode Extension)
**주 역할**: 기존 코드 분석 + 코드 작성 + DB 마이그레이션 실행

- 기존 코드를 직접 읽고 패턴을 파악한 뒤 작업
- DB 스키마 변경 + Supabase API 실행
- 복잡한 비즈니스 로직 구현
- 빌드 오류 디버깅

### 1-2. 외부 AI (Claude.ai, Cursor, Copilot 등)
**주 역할**: PRD 기반 신규 모듈 독립 개발

- PRD 문서와 공통 룰을 받아서 독립 모듈 구현
- 새 페이지 / 새 API 라우트 생성
- 기존 코드 수정 최소화 (기존 파일 읽지 않고도 작업 가능하도록 PRD 작성)

## 3. 모듈별 담당 AI 배정 가이드

각 Phase의 작업은 독립적으로 분리되어 있어 **여러 AI가 동시에 작업 가능**.

### 동시 작업 가능 조합 (예시)

```
Claude Code:     P0-1 AdminRole 마이그레이션 + P0-2 MemberProfile DB
외부 AI A:       P0-3 Course 마스터 CRUD UI
외부 AI B:       L-1 수강계약서 PDF 생성 로직
```

### 충돌 방지 원칙

| 규칙 | 내용 |
|---|---|
| 파일 분리 | 같은 파일을 두 AI가 동시에 수정하지 않는다 |
| API 네임스페이스 | 각 AI가 작업하는 /api/[경로]는 겹치지 않게 배정 |
| DB 스키마 | 스키마 변경은 한 번에 한 AI만 (Claude Code 담당) |
| 공통 컴포넌트 | `/components/ui/`는 Claude Code만 수정 |

## 6. 코드 리뷰 체크리스트

외부 AI가 코드를 제출했을 때 Claude Code가 검토하는 항목:

**[최우선 — 00_개발공통룰.md §0 핵심 데이터 연동 원칙]**
- [ ] 학생 목록/상세 화면에 학번·이름·연락처·수강내역 4개가 모두 표시됨
- [ ] 학생명/학번 클릭 시 `/admin/students/[examNumber]`로 이동하는 링크 있음
- [ ] API 응답에 `student: { examNumber, name, mobile, currentEnrollments }` 포함됨
- [ ] Prisma 조회 시 student include에 name·mobile·enrollments 포함됨
- [ ] 학생 검색 입력이 학번·이름·연락처 3가지 모두 검색 가능

**[기본 규칙]**
- [ ] 00_개발공통룰.md 전체 준수 여부
- [ ] TypeScript strict 오류 없음
- [ ] 인증 체크 누락 없음 (`getAdminSession()`)
- [ ] SQL Injection 위험 없음 (Prisma ORM 사용)
- [ ] 환경 변수 클라이언트 노출 없음
- [ ] DB 레코드 삭제 코드 없음 (금전 관련 — 논리 삭제만 허용)

**[비즈니스 규칙]**
- [ ] 수납 항목 카테고리 코드 (`PaymentCategory` enum, §8) 올바르게 사용
- [ ] `examNumber` = 학원 자동채번 학번 (공무원 시험 수험번호와 혼동 금지)
- [ ] 수강번호(`enrollmentNumber`) = YY-NNNN 형식으로 자동 채번
- [ ] 미납/환불 관련 기능은 종합반(`TUITION_COMPREHENSIVE`)에만 적용
- [ ] 특강 강사 배분율 수정 = 원장(DIRECTOR) 권한만 가능

## 10. 멀티테넌시 필수 적용 사항 (2026-03-21 추가)

> 지점(Academy) 구조로 전환됨에 따라 모든 신규 개발에서 반드시 준수해야 한다.
> 상세 규칙: `00_개발공통룰.md §14`

### 10-1. API 라우트 표준 패턴 (변경)

```typescript
// ✅ 변경 후 — 모든 API 라우트에서 getAdminContext() 사용
import { getAdminContext } from '@/lib/auth'

export async function GET() {
  const ctx = await getAdminContext()
  if (!ctx) return Response.json({ error: '인증이 필요합니다.' }, { status: 401 })

  const data = await prisma.student.findMany({
    where: { academyId: ctx.activeAcademyId ?? ctx.academyId ?? undefined }
  })
  return Response.json({ data })
}

// ❌ 기존 패턴 — 더 이상 사용 금지
import { getAdminSession } from '@/lib/auth'
const session = await getAdminSession()
const data = await prisma.student.findMany()  // academyId 필터 없음
```

### 10-2. 설정 조회 패턴 (지점별 독립)

```typescript
// ✅ 지점별 시험 과목 조회 (Subject enum 직접 사용 금지)
const subjects = await prisma.examSubject.findMany({
  where: { academyId: ctx.activeAcademyId ?? ctx.academyId, isActive: true },
  orderBy: { displayOrder: 'asc' }
})
```

### 10-3. 외부 AI에게 추가로 전달할 컨텍스트

신규 기능 개발 지시 시 아래 항목을 반드시 포함:

```
추가 읽기:
4. 멀티지점_운영전환_기획서.md  ← 지점 구조, academyId 필터 규칙
5. 00_개발공통룰.md §14         ← 멀티테넌시 코드 패턴

핵심 규칙:
- 모든 DB 조회에 academyId 필터 적용
- getAdminContext() 사용 (getAdminSession() 금지)
- 신규 레코드 생성 시 academyId 필수 포함
- 설정 조회는 academySetting 테이블 사용 (하드코딩 금지)
- 시험 과목은 examSubject 테이블 사용 (Subject enum 직접 사용 금지)
```

### 10-4. SUPER_ADMIN 분기 처리

```typescript
// SUPER_ADMIN이 전체 집계 모드일 때 처리
const whereClause = ctx.isSuperAdmin && !ctx.activeAcademyId
  ? {}                                    // 전체 조회
  : { academyId: ctx.activeAcademyId ?? ctx.academyId }

const data = await prisma.payment.findMany({ where: whereClause })
```
