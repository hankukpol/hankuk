## 0. 핵심 데이터 연동 원칙 (⚠️ 절대 규칙 — 예외 없음)

> **시스템의 모든 페이지, 모든 기능에서 학생 4대 핵심 데이터는 반드시 연동되어야 한다.**

### 0-1. 학생 4대 핵심 데이터

| 데이터 | 필드명 | 설명 |
|---|---|---|
| **학번(수험번호)** | `examNumber` | 학원 자동채번, 시스템 전체 primary key |
| **이름** | `name` | 원생 이름 |
| **연락처** | `mobile` | 원생 휴대전화 |
| **수강내역** | `enrollments[]` | 종합반 + 특강 수강 이력 전체 |

### 0-2. 연동 의무 사항

```
모든 기능(성적, 출결, 수납, 상담, 포인트, 모의고사, 사유서 등)에서:

1. 학생 데이터 표시 시 반드시 학번·이름·연락처를 함께 표시
2. 어떤 화면에서든 학생명/학번 클릭 시 학생 상세 페이지(/admin/students/[examNumber])로 이동
3. 수강내역 없는 학생에게 성적/출결/면담 등 데이터를 연결할 때 수강내역 경고 표시
4. 학생 검색 시 학번·이름·연락처 3가지 모두 검색 가능
5. 수강내역 변경(등록/퇴원/환불) 시 관련된 모든 기능(성적, 출결, 포인트 등)에 즉시 반영
```

### 0-3. UI 표준 패턴

```tsx
// ✅ 모든 목록 화면에서 학생 표시 표준
// 학번·이름·연락처는 항상 함께 — 클릭 시 학생 상세로 이동
<StudentCell
  examNumber="81697"
  name="홍길동"
  mobile="010-1234-5678"
  href={`/admin/students/81697`}
/>

// ✅ 모든 상세 화면 상단에 학생 요약 카드 표시
<StudentSummaryCard
  examNumber={student.examNumber}
  name={student.name}
  mobile={student.mobile}
  enrollments={student.enrollments}  // 수강내역 요약
/>
```

### 0-4. API 응답 표준

```typescript
// ✅ 학생이 포함된 모든 API 응답에 4대 핵심 데이터 포함
interface StudentBasic {
  examNumber: string    // 학번
  name: string          // 이름
  mobile: string | null // 연락처
  currentEnrollments: { // 현재 수강내역 (간략)
    type: string        // "공채 12개월 종합반"
    status: string      // "ACTIVE"
    cohortName?: string // "52기"
  }[]
}

// ❌ 금지 — 학번만 있고 이름/연락처/수강내역 없는 응답
{ scoreId: "xxx", examNumber: "81697", score: 85 }

// ✅ 올바른 응답 — 학생 기본 정보 반드시 포함
{ scoreId: "xxx", student: { examNumber: "81697", name: "홍길동", mobile: "...", currentEnrollments: [...] }, score: 85 }
```

### 0-5. DB 조회 표준

```typescript
// ✅ 학생 관련 조회 시 항상 기본 정보 include
const scores = await prisma.examScore.findMany({
  where: { sessionId },
  include: {
    student: {
      select: {
        examNumber: true,
        name: true,
        mobile: true,
        enrollments: {
          where: { status: 'ACTIVE' },
          select: { courseType: true, cohort: { select: { name: true } } }
        }
      }
    }
  }
})

// ❌ 금지 — 학생 정보 없이 examNumber만 조회
const scores = await prisma.examScore.findMany({ where: { sessionId } })
```

## 6. DB 규칙 (Prisma + Supabase)

### 6-1. 마이그레이션 방법 (⚠️ 중요)

> **이 프로젝트는 `prisma db push` 직접 실행 불가** (IPv6 문제)

```bash
# 스키마 변경 시:
# 1. prisma/schema.prisma 수정
# 2. SQL 생성
npx prisma migrate diff --from-empty --to-schema-datamodel prisma/schema.prisma --script

# 3. Supabase Management API로 실행
POST https://api.supabase.com/v1/projects/psfsprodoedjjngldyzr/database/query
Authorization: Bearer [API_KEY]
Content-Type: application/json
{"query": "SQL 내용"}
```

### 6-2. 모델 네이밍 규칙

```prisma
model ExampleModel {
  id          String    @id @default(cuid())
  examNumber  String    // FK to Student
  createdAt   DateTime  @default(now())
  updatedAt   DateTime  @updatedAt

  @@map("example_models")  // DB 테이블명: snake_case
}
```

### 6-3. 절대 금지 사항

- 기존 컬럼 `@id`, `examNumber` 타입 변경 금지
- 기존 테이블 삭제 금지 (마이그레이션으로 추가만 가능)
- `@default(autoincrement())` 새 테이블에 사용 금지 → `@default(cuid())` 사용
- 금전 관련 레코드 `DELETE` 금지 → status 필드로 논리 삭제

### 6-4. 금전 데이터 불변 원칙

```
결제/수납 레코드는 절대 삭제하지 않는다.
취소 = 새로운 CANCELLED 상태 레코드 추가
환불 = 새로운 Refund 레코드 추가
수정 = 원본 유지 + 새 버전 레코드 추가
```

## 8. 수납 항목 및 수익 분류 (비즈니스 규칙 — 코드 반드시 준수)

> 모든 수납/정산 기능은 이 분류를 따른다.
> 수납 화면에서 "항목 추가" 버튼의 카테고리 코드와 정산 방식이 1:1 대응된다.

| 카테고리 | 코드 | 수납 버튼명 | 설명 | 정산 방식 |
|---|---|---|---|---|
| 종합반 수강료 | `TUITION_COMPREHENSIVE` | 수강료 추가 | 기간제 종합반 수강료 | 학원 전체 수익 |
| 특강 단과 수강료 | `TUITION_SPECIAL_LECTURE` | 수강료 추가 | 특강 단과 수강료 | 강사 배분 % + 학원 % |
| 교재 판매 | `SALES_TEXTBOOK` | 교재비 추가 | 교재 판매 (대여 아님) | 별도 수익 |
| 교구·소모품 | `SALES_MATERIAL` | 교구비 추가 | 교구·소모품 판매 | 별도 수익 |
| 원서 대행비 | `APPLICATION_FEE` | 원서비 추가 | 시험 원서 접수 대행 | 별도 수익 |
| 사물함 이용료 | `FACILITY_LOCKER` | 기타 추가 | 사물함 월별 이용료 | 별도 수익 |
| 스터디룸 | `FACILITY_STUDY_ROOM` | 기타 추가 | 시간당 1,000원 | 별도 수익 |
| 기타 | `ETC` | 기타 추가 | 기타 수납 | 별도 수익 |

## 12. 원생 식별자 규칙 (⚠️ 중요 — 혼동 주의)

```
학번 = 수험번호 = examNumber  (모두 동일한 개념)
  → 학원이 원생 최초 등록 시 자동 채번 (예: 81697)
  → 순차 증가 방식, 변경 불가, 퇴원 후 재등록 시에도 유지
  → 시스템 전체 primary key (Student.examNumber)

⚠️ 공무원 시험 수험번호와 완전히 다른 개념:
  → 공무원 시험 수험번호 = 시험 응시 시 발급되는 별도 번호
  → Student 모델의 officialExamNumber 필드로 별도 관리 (선택 입력)

수강번호 = enrollmentNumber  (수강 등록마다 부여)
  → 형식: YY-NNNN (예: 26-0101, 26-0102, ...)
  → YY: 등록 연도 뒤 2자리 / NNNN: 해당 연도 수강 등록 순차 번호
  → 학번(examNumber)과 완전히 별개
  → 취소·환불 후에도 번호 유지 (삭제 불가)
```

## 13. 핵심 비즈니스 규칙 요약

```
1.  학번(examNumber) = 학원 자동채번, 시스템 전체 식별자 (§12 참조)
2.  수강번호(enrollmentNumber) = YY-NNNN, 수강 등록마다 자동 채번 (§12 참조)
3.  종합반 = 공채/경채 × 기간(개월) × 기수(期數) 조합 → 학원 전체 수익
4.  특강 단과 = 강사별 수익 배분율(%) 설정 필수, 원장만 수정 가능
5.  기수 = 4개월 단위 코호트, 수험유형별 독립 운영 (공채 52기, 경채 52기 등)
6.  교재 = 판매만 (대여 없음), 별도 재고/정산
7.  스터디룸 = 시간당 1,000원 기본값
8.  수납 레코드는 삭제 불가 (취소 = CANCELLED 상태, 환불 = 새 Refund 레코드)
9.  모든 금전 처리에 처리 직원 + 시각 기록 필수
10. 학원법 §14 수강계약서 = 종합반 수강 등록 즉시 자동 생성
11. 학원법 §18 환불 = 종합반 전용 (특강/교재는 별도 정책)
      - 수강 전: 전액 환불
      - 1/3 경과 전: 납부액의 2/3 환불
      - 1/2 경과 전: 납부액의 1/2 환불
      - 1/2 경과 후: 환불 없음
12. 미납자/환불자 관리 = 종합반 전용 (특강 단과에 적용하지 않음)
13. 합격자 = 필기합격/최종합격 구분, 연도·차수별 관리
```

## 14. 멀티테넌시 규칙 (⚠️ 2026-03-21 추가 — 반드시 준수)

> **모든 데이터 조회·생성·수정은 academyId(지점) 컨텍스트를 반드시 포함해야 한다.**
> 이 규칙을 어기면 다른 지점의 데이터가 혼재·노출될 수 있다.

### 14-1. 핵심 원칙

```
원칙 1 — 코드 단일화
  코드 수정은 모든 지점에 동일하게 반영된다. 지점별 코드 분기 금지.

원칙 2 — 설정 독립화
  학원 기본정보, 시험 과목, 사물함 배치, 할인 정책 등 모든 설정은
  지점(academyId)별로 독립 저장·적용된다.

원칙 3 — 데이터 격리
  모든 학생/결제/성적/출결 데이터는 academyId로 격리된다.
  SUPER_ADMIN만 전 지점 데이터에 접근할 수 있다.
```

### 14-2. 개발자 코드 패턴

```typescript
// ✅ 올바른 패턴 — 모든 API에서 academyId 필터 적용
import { getAdminContext } from '@/lib/auth'

const ctx = await getAdminContext()
if (!ctx) return Response.json({ error: '인증이 필요합니다.' }, { status: 401 })

// academyId가 null이면 SUPER_ADMIN → 전체 조회 (또는 선택된 지점만)
const students = await prisma.student.findMany({
  where: ctx.academyId ? { academyId: ctx.academyId } : undefined
})

// ❌ 금지 — academyId 없이 조회 (모든 지점 데이터가 섞임)
const students = await prisma.student.findMany()
```

### 14-3. 신규 데이터 생성 시

```typescript
// ✅ 모든 신규 레코드에 academyId 포함
await prisma.student.create({
  data: {
    ...studentData,
    academyId: ctx.academyId,  // 반드시 포함
  }
})

// ❌ 금지 — academyId 없는 생성
await prisma.student.create({ data: studentData })
```

### 14-4. SUPER_ADMIN 특별 처리

```typescript
// SUPER_ADMIN은 지점 전환 쿠키를 기반으로 activeAcademyId 사용
// activeAcademyId = null → 전체 조회 모드 (집계·통계 등에서 활용)
// activeAcademyId = 숫자 → 해당 지점만 조회

interface AdminContext {
  userId: string
  adminUser: AdminUser
  academyId: number | null   // 소속 지점 (SUPER_ADMIN이면 null)
  isSuperAdmin: boolean
  activeAcademyId: number | null  // 현재 활성 지점 (SUPER_ADMIN 전환 기준)
}
```

### 14-5. 설정 조회 패턴

```typescript
// ✅ 지점별 설정 조회 — 항상 academyId 기준
const settings = await prisma.academySetting.findUnique({
  where: { academyId: ctx.activeAcademyId ?? ctx.academyId }
})

// ✅ 지점별 시험 과목 조회 (하드코딩 Subject enum 사용 금지)
const subjects = await prisma.examSubject.findMany({
  where: { academyId: ctx.activeAcademyId ?? ctx.academyId, isActive: true },
  orderBy: { displayOrder: 'asc' }
})
```

### 14-6. 접근 범위 정리

| 역할 | 접근 범위 | 설정 변경 |
|------|-----------|-----------|
| **SUPER_ADMIN** | 전 지점 데이터 + 지점 생성·삭제 | 모든 지점 설정 가능 |
| DIRECTOR (원장) | 소속 지점만 | 소속 지점 설정 가능 |
| 나머지 역할 | 소속 지점만 | 권한별 제한 (10_직원권한_PRD.md §2 참조) |
