# 한국경찰학원 통합 관리 시스템 — Codex 운영 규칙

## 0. 목적
이 문서는 이 저장소에서 작업하는 모든 Codex, 팀 에이전트, 서브에이전트의 강제 운영 규칙이다.
이 문서보다 우선하는 프로젝트 규칙은 `개발계획/` 내부 PRD와 명시적 사용자 지시뿐이다.

핵심 원칙:
- 모든 개발 작업은 반드시 `개발계획` 폴더의 PRD를 먼저 확인한 뒤 시작한다.
- 문서 확인 없이 구현부터 시작하는 것을 금지한다.
- 팀 에이전트 병렬 작업 시에도 같은 규칙을 그대로 적용한다.
- `개발계획/` 문서는 읽기 전용이며 수정하지 않는다.

## 1. 프로젝트 구조

```text
/                    프로젝트 루트
├── prisma/schema.prisma
├── src/app/admin/   관리자 페이지
├── src/app/student/ 학생 포털
├── src/app/api/     API 라우트
├── src/lib/         서비스 로직
├── src/components/  컴포넌트
└── 개발계획/         PRD 문서 폴더 (수정 금지)
```

## 2. 작업 전 필수 문서 확인
모든 작업은 아래 순서를 강제로 따른다.

1. `개발계획/00_개발공통룰.md`
2. `개발계획/01_마스터플랜.md`
3. 해당 기능 PRD
4. `한국경찰학원 통합관리 시스템 — AI 개발 인수인계 지침서.txt`

예시:
- 수강/수납 작업: `개발계획/04_수납결제_정산_PRD.md`
- 학생 포털 작업: `개발계획/12_대시보드_알림_PRD.md` 및 학생 포털 관련 PRD
- 직원/설정 작업: `개발계획/10_직원권한_PRD.md`
- 시설 작업: 마스터플랜의 시설 섹션 + 관련 네비게이션/UI 설계 문서

작업 시작 전 체크리스트:
- 이번 작업에서 읽은 PRD 파일 경로를 명시한다.
- 문서상 요구 경로와 현재 실제 경로가 같은지 먼저 확인한다.
- 기존 화면이나 API를 재사용할 수 있는지 먼저 확인한다.
- 문서와 현재 구현이 다르면 구현 전에 차이를 정리한다.

## 3. 팀 에이전트 운영 표준
팀 에이전트 작업은 아래 구조를 기본값으로 사용한다.

### 3-1. 역할
- 리드 에이전트: PRD 확인, 작업 분해, 충돌 방지, 최종 통합, 최종 검증 담당
- Worker 에이전트: 실제 구현 담당
- Explorer 에이전트: 리뷰 및 문서 정합성 검증 담당

### 3-2. 기본 편성
중간 이상 작업은 아래 구조를 기본으로 한다.
- Worker 1개 이상: 구현
- Explorer 1개 이상: PRD 대조 리뷰
- 리드 에이전트 1개: 통합 및 검증

### 3-3. 리뷰 에이전트 필수 점검 항목
Explorer 리뷰 에이전트는 반드시 아래를 확인한다.
- 작업 전에 읽어야 할 PRD를 실제로 읽었는지
- 문서상 라우트와 실제 라우트가 일치하는지
- 학생 4대 데이터 규칙이 적용되어야 하는 화면인지
- 인증 방식이 규칙에 맞는지
- API 응답 형식이 규칙에 맞는지
- Prisma ORM만 사용했는지
- 기존 화면 재사용이 가능한데 중복 구현하지 않았는지
- 구현 우선순위가 `수강 등록+수납 > 성적·출결 > 포털·시설`을 따르는지

### 3-4. 병렬 작업 규칙
- Worker끼리 파일 소유 범위를 명확히 분리한다.
- 같은 파일을 여러 Worker가 동시에 수정하지 않는다.
- 경로 alias 작업은 기존 구현을 재사용하는 방향을 우선한다.
- 스키마 변경이 필요한 작업은 일반 Worker가 아니라 관련 Prisma 스킬 흐름으로 분리한다.

## 4. 개발 우선순위
항상 아래 우선순위를 유지한다.

1. 수강 등록 + 수납
2. 성적 + 출결
3. 포털 + 시설

문서상 공백이 여러 개여도, 우선순위가 높은 축을 먼저 닫는다.

## 5. 핵심 비즈니스 규칙

### 5-1. 학생 4대 데이터
모든 관리자 화면과 학생 중심 화면에서 아래 데이터가 요구되면 함께 보여준다.
- `examNumber` — 학번
- `name` — 이름
- `mobile` — 연락처 (`phone` 필드는 화면에서 `mobile` 의미로 사용)
- `enrollments[]` — 수강내역

학생명 또는 학번 클릭 시:
- `/admin/students/[examNumber]` 이동 필수

### 5-2. 인증
관리자 API와 관리자 페이지는 기본적으로 아래 인증 규칙을 따른다.

```typescript
import { getAdminSession } from '@/lib/auth'

const session = await getAdminSession()
if (!session) {
  return Response.json({ error: 'Unauthorized' }, { status: 401 })
}
```

### 5-3. API 응답 형식

```typescript
// 성공
return Response.json({ data: result })

// 오류
return Response.json({ error: '메시지' }, { status: 400 })
```

### 5-4. DB 접근

```typescript
import { prisma } from '@/lib/prisma'
```

규칙:
- Prisma ORM만 사용한다.
- raw SQL은 금지한다.
- 스키마 변경은 별도 Prisma 배포 규칙을 따른다.

## 6. 수정 금지 / 전담 파일
외부 Worker 또는 일반 에이전트는 아래 파일을 직접 수정하지 않는다.
- `prisma/schema.prisma`
- `src/lib/auth.ts`
- `src/lib/prisma.ts`
- `src/components/ui/*`

이 범위는 리드 에이전트 또는 Prisma/공용 인프라 전담 흐름에서만 다룬다.

## 7. 스킬 사용 기준
가능하면 아래 스킬을 작업 범위에 맞게 사용한다.
- 학생 포털, 학생 상세, 목표/이력/출결/오답노트: `morning-mock-student-lifecycle`
- 관리자 운영 화면, 공지/설정/사이트 운영, 관리자 CRUD: `police-admin-site-ops`
- Prisma 스키마, 마이그레이션, 배포 민감 작업: `police-prisma-deploy`
- 점수 계산, 합격예측, 컷라인: `police-scoring-prediction`
- 제출/결과 흐름: `police-submission-result-flow`
- 분석/리포트/캘린더/대시보드: `morning-mock-analytics-reporting`

스킬 사용 규칙:
- 작업 범위가 스킬 설명과 명확히 맞으면 해당 스킬을 우선 적용한다.
- 스킬보다 PRD가 우선이다.
- 스킬은 구현 가이드이며, 비즈니스 규칙의 기준은 항상 `개발계획` 문서다.

## 8. 작업 절차 표준
모든 구현은 아래 절차를 기본값으로 한다.

### 8-1. 착수 전
- PRD 읽기
- 현재 구현 확인
- 문서 경로와 실제 경로 비교
- 재사용 가능한 화면/API 탐색
- 팀 에이전트 분할 여부 결정

### 8-2. 구현 중
- 구현 전에 무엇을 바꾸는지 간단히 공유한다.
- 병렬 작업이면 Worker마다 파일 소유 범위를 고정한다.
- PRD에 없는 새로운 구조를 만들기 전에 기존 구조를 우선 활용한다.

### 8-3. 구현 후
최소 검증:
- `pnpm run typecheck`

필요 시 추가 검증:
- `pnpm run lint`
- 기능별 검증 스크립트
- `pnpm run build`

종료 보고에는 아래를 포함한다.
- 읽은 PRD 문서
- 실제 수정한 경로
- 문서 대비 어떤 공백을 메웠는지
- 검증 결과
- 남은 위험 또는 후속 작업

## 9. 문서 경로와 실제 경로가 다를 때의 규칙
문서상 기능은 있는데 실제 구현 경로가 다를 때는 아래 순서를 따른다.

1. 기존 구현이 있으면 우선 재사용한다.
2. 가능하면 얇은 alias route를 추가해 문서 경로를 맞춘다.
3. 중복 화면 구현은 피한다.
4. 문서와 실제가 크게 다르면 최종 보고에서 차이를 명시한다.

## 10. 개발 문서 수정 금지
`개발계획/` 내부 문서는 프로젝트 기준 문서다.
규칙:
- 읽기만 가능
- 임의 수정 금지
- 구현이 문서와 다르면 문서를 바꾸지 말고 코드 또는 alias로 정합성을 맞춘다
