## 1. 소방단기 과락 처리 방식 (역산 분석)

소방단기는 과락 처리 방식을 공개하지 않으므로, **API 데이터를 역산**하여 추정.

### 1.1 경쟁률 역산 — 과락 제외 강력 증거

소방단기 API에서 `경쟁률 = cnt(입력인원) / nowSelect(선발인원)`이 맞는지 검증:

| 지역 | 채용구분 | 입력인원(cnt) | 선발(nowSelect) | cnt/nowSelect | 실제 경쟁률 | 추정 유효인원 | 추정 과락자 | 과락률 |
|------|---------|-------------|----------------|---------------|-----------|-------------|-----------|--------|
| 경기 | 공채(남) | 982 | 319 | **3.08** | **2.55** | 813 | 169 | **17.2%** |
| 서울 | 공채(남) | 560 | 221 | **2.53** | **2.04** | 451 | 109 | **19.5%** |
| 경남 | 공채(남) | 401 | 145 | **2.77** | **2.40** | 348 | 53 | **13.2%** |
| 대구 | 공채(남) | 221 | 65 | **3.40** | **2.78** | 181 | 40 | **18.1%** |
| 경기 | 구급(여) | 358 | 125 | **2.86** | **2.35** | 294 | 64 | **17.9%** |

경쟁률 기준 TOP5에서도 동일 패턴:

| 지역 | 채용구분 | cnt | nowSelect | cnt/nowSelect | 실제 경쟁률 | 과락률 |
|------|---------|-----|-----------|---------------|-----------|--------|
| 경기 | 공채(여) | 38 | 6 | 6.33 | 5.17 | **18.4%** |
| 광주 | 관련학과(남) | 12 | 2 | 6.00 | 5.00 | **16.7%** |
| 충북 | 관련학과(여) | 5 | 1 | 5.00 | 5.00 | **0%** (전원 통과) |
| 경북 | 공채(여) | 45 | 8 | 5.63 | 4.50 | **20.0%** |
| 경기 | 구조(남) | 75 | 15 | 5.00 | 4.27 | **14.7%** |

### 1.2 역산 결론

```
경쟁률 공식 = (입력인원 - 과락자) / 선발인원
           = 유효인원 / 선발인원

유효인원 = competitionRate × nowSelect
과락자수 = cnt - 유효인원
과락률   = 과락자수 / cnt × 100
```

**과락률이 0~20% 범위로 매우 일관적** → **과락자 제외 경쟁률**이 거의 확실.
(충북 관련학과(여) 5명은 전원 과락 없어서 cnt/nowSelect=경쟁률이 일치)

### 1.3 소방단기 각 지표별 과락 처리 추정

| 지표 | 과락 포함? | 근거 |
|------|----------|------|
| **입력인원 (cnt)** | **포함** | TOP5 테이블에 전체 입력자 수로 표시 |
| **경쟁률 (competitionRate)** | **제외** | 역산으로 확인 (위 표 참조) |
| **1배수컷 (oneMultipleCut)** | **제외** | 과락자는 불합격이므로 순위에서 제외해야 의미 있음 |
| **입력자 평균 (averageScore)** | **제외 추정** | 공채 평균 186점 = 300점 만점의 62%. 과락기준 총점 180점(60%)과 근접. 과락자 포함 시 평균이 더 낮아야 자연스러움 |
| **합격 확실권 (highestScore)** | **제외** | 1배수컷 기반 산출이므로 당연히 과락 제외 |
| **합격선 (passCut)** | **제외** | 최고점(1등 점수)일 가능성. 과락자가 1등일 수 없으므로 제외 |

### 1.4 passCut 필드의 정체

```
경기 공채(남):
  passCut: 236.00 > highestScore: 212.00 > oneMultipleCut: 208.00 > averageScore: 186.89

passCut이 항상 가장 높음 → "합격선"이 아니라 실제 "1등 점수(최고점)"일 가능성 높음
```

| 데이터 | passCut | highestScore | oneMultipleCut | averageScore |
|--------|---------|-------------|----------------|-------------|
| 경기 공채(남) | 236.00 | 212.00 | 208.00 | 186.89 |
| 서울 공채(남) | 232.00 | 220.00 | 200.00 | 185.68 |
| 경남 공채(남) | 240.00 | 208.00 | 204.00 | 188.47 |
| 대구 공채(남) | 228.00 | 216.00 | 212.00 | 182.38 |
| 경기 구급(여) | 165.00 | 140.00 | 140.00 | 133.22 |

**추정 해석**:
- `passCut` = **전체 최고점** (1등의 점수)
- `highestScore` = **확실권 하한선** (이 이상이면 합격 확실, UI에 "합격 확실권"으로 표시)
- `oneMultipleCut` = **1배수 커트라인** (선발인원 N등의 점수)
- `averageScore` = **과락 제외 평균점수**

## 2. 우리 프로젝트 과락 처리 방식 (코드 분석)

### 2.1 핵심 WHERE 조건 2가지

우리 코드에는 과락 필터링을 위한 **2가지 WHERE 조건**이 존재:

```typescript
// (A) 과락 제외 — 순위/평균/합격컷 산출용
const populationWhere = {
  examId: activeExam.id,
  isSuspicious: false,
  subjectScores: {
    some: {},                    // 과목점수가 하나라도 있고
    none: { isFailed: true },   // 과락 과목이 하나도 없는 제출만
  },
};

// (B) 과락 포함 — 전체 참여인원 표시용
const allParticipantCountWhere = {
  examId: activeExam.id,
  isSuspicious: false,
  subjectScores: { some: {} },  // 채점된 제출이면 모두 포함
};
```

### 2.2 각 API별 과락 처리 상세

#### 2.2.1 메인 통계 API (`/api/main-stats`) — `main-stats/route.ts`

| 지표 | 과락 포함? | 코드 위치 | 사용 WHERE |
|------|----------|---------|-----------|
| **참여인원 (participantCount)** | **포함** | 658~665줄 | `allParticipantCountWhere` (B) |
| **평균점수 (averageFinalScore)** | **제외** | 647~657줄 | `populationWhere` (A) |
| **1배수컷 (oneMultipleCutScore)** | **제외** | 666~673줄 scoreBandStats | `populationWhere` (A) |
| **확실권 (sureMinScore)** | **제외** | scoreBands 기반 | `populationWhere` (A) |
| **유력권 (likelyRange)** | **제외** | scoreBands 기반 | `populationWhere` (A) |
| **가능권 (possibleRange)** | **제외** | scoreBands 기반 | `populationWhere` (A) |
| **경쟁률 (competitionRate)** | **해당없음** | 773~776줄 | `출원인원/선발인원` (과락과 무관) |
| **점수 분포도 (총점)** | **포함** | 675~684줄 | 과락 필터 없음 |
| **점수 분포도 (과목별)** | **포함** | 688~700줄 | 과락 필터 없음 |

**핵심 설계**:
```
참여인원 = 과락 포함 (전체 입력자 수)
평균점수 = 과락 제외 (유효 경쟁자의 평균)
합격컷   = 과락 제외 (유효 경쟁자 순위 기반)
분포도   = 과락 포함 (전체 점수 분포를 보여줌)
```

#### 2.2.2 성적분석 API (`/api/result`) — `result/route.ts`

**자기 과락 여부에 따라 비교 모집단이 달라짐**:

```typescript
// result/route.ts 97~111줄
function getPopulationConditionSql(submissionHasCutoff: boolean) {
  if (submissionHasCutoff) {
    // 자기가 과락이면 → 전체(과락 포함)에서 순위
    return Prisma.sql`AND s."isSuspicious" = false`;
  }

  // 자기가 과락 아니면 → 과락 제외에서 순위
  return Prisma.sql`
    AND s."isSuspicious" = false
    AND NOT EXISTS (
      SELECT 1 FROM "SubjectScore" sf
      WHERE sf."submissionId" = s.id AND sf."isFailed" = true
    )
  `;
}
```

| 내 상태 | 비교 모집단 | 이유 |
|--------|-----------|------|
| **과락 아님** | 과락 제외 모집단 | 유효 경쟁자끼리만 비교 (의미 있는 순위) |
| **과락임** | 전체 모집단 (과락 포함) | 과락자가 비과락 모집단에 포함되면 순위가 불공정 |

이 설계에서 표시되는 항목:

| 지표 | 과락 포함? | 조건 |
|------|----------|------|
| **내 석차 (totalRank)** | 내 상태에 따라 다름 | 과락 아니면 제외, 과락이면 포함 |
| **전체인원 (totalParticipants)** | 내 상태에 따라 다름 | 위와 동일 |
| **과목별 석차** | 내 상태에 따라 다름 | 위와 동일 |
| **백분위 (percentile)** | 내 상태에 따라 다름 | 위와 동일 |
| **상위% (topPercent)** | 내 상태에 따라 다름 | 위와 동일 |
| **평균점수 (총점/과목별)** | 내 상태에 따라 다름 | 위와 동일 |
| **rankingBasis 표시** | — | `"NON_CUTOFF_PARTICIPANTS"` or `"ALL_PARTICIPANTS"` |

## 3. 소방단기 vs 우리 프로젝트 비교 요약

| 지표 | 소방단기 (추정) | 우리 프로젝트 | 일치? |
|------|---------------|-------------|------|
| **참여인원** | **과락 포함** (cnt 전체) | **과락 포함** (allParticipantCountWhere) | **일치** |
| **경쟁률** | **과락 제외** (역산 확인) | 출원/선발 기반 (입력자 기반 아님) | **다름** (아래 참고) |
| **1배수컷** | **과락 제외** | **과락 제외** | **일치** |
| **평균점수** | **과락 제외** (추정) | **과락 제외** | **일치** |
| **확실권** | **과락 제외** | **과락 제외** | **일치** |
| **석차 (성적분석)** | 불명 (로그인 필요) | **조건부** (내 과락 여부에 따라) | — |
| **점수 분포도** | 불명 | **과락 포함** | — |

### 경쟁률 계산 방식의 차이

```
소방단기: 경쟁률 = (입력인원 - 과락자) / 선발인원  ← 입력자 기반, 과락 제외
우리:     경쟁률 = 출원인원 / 선발인원              ← 공식 응시인원 기반, 과락 무관
```

소방단기는 **서비스 입력자 기반 경쟁률**을 보여주고,
우리는 **실제 원서접수 인원 기반 경쟁률**을 보여줌.

## 4. 우리 프로젝트 개선 고려사항

### 4.1 현재 잘 되어 있는 것

- 참여인원은 과락 포함 → 서비스 규모를 정확히 보여줌
- 평균/합격컷은 과락 제외 → 유효 경쟁만 반영
- 성적분석 석차는 내 과락 여부에 따라 조건부 → 공정한 순위 제공
- `rankingBasis` 표시 → 사용자에게 순위 기준 명시

### 4.2 참고할 소방단기 방식

#### (1) 입력자 기반 경쟁률 추가 표시

현재 우리는 `출원인원/선발인원`만 표시. 소방단기처럼 **입력자 기반 경쟁률**도 추가하면 유용:

```
공식 경쟁률: 745명 / 65명 = 11.46:1  (출원 기준)
실시간 경쟁률: 181명 / 65명 = 2.78:1  (과락 제외 입력자 기준)
```

**구현 방법**:
```typescript
// main-stats/route.ts에서 과락 제외 participantCount 활용
const validParticipantCount = participantStats의 _count._all;  // 이미 있음
const inputBasedCompetitionRate = validParticipantCount / recruitCount;
```

#### (2) 참여인원 옆에 과락자 수/비율 표시

```
참여인원: 221명 (과락 40명, 유효 181명)
```

**구현 방법**:
```
과락자수 = allParticipantCount - validParticipantCount
(두 쿼리의 차이, 이미 데이터 있음)
```

#### (3) 과락률 통계 제공

관리자 대시보드나 메인 페이지에 과락률 표시:
```
전체 과락률: 18.1% (40/221명)
  - 과목별 과락: 15.4% (34명)
  - 총점 과락: 2.7% (6명)
```

## 6. 과락 처리 흐름도

```
사용자 OMR 제출
  │
  ├─ scoring.ts: 과목별 채점
  │   ├─ 과목별 40% 미만? → SubjectScore.isFailed = true
  │   └─ 총점 60% 미만?   → 모든 SubjectScore.isFailed = true
  │
  ├─ Submission 저장 (totalScore, finalScore)
  │
  ├─ [메인 페이지 표시]
  │   ├─ 참여인원: 과락 포함 COUNT
  │   ├─ 평균점수: 과락 제외 AVG (isFailed=true인 제출 제외)
  │   ├─ 1배수컷:  과락 제외 순위 기반
  │   └─ 점수분포: 과락 포함 (전체 분포)
  │
  ├─ [성적분석 페이지]
  │   ├─ 내가 과락 아님 → 과락 제외 모집단에서 석차/백분위
  │   └─ 내가 과락임   → 전체 모집단에서 석차/백분위
  │
  └─ [합격컷 자동발표]
      ├─ 참여율: 과락 제외 인원 / (모집인원 × 배수)
      ├─ 1배수컷: 과락 제외 순위 기반
      └─ 등급(확실/유력/가능): 과락 제외 순위 기반
```
