# 새 프로젝트 추가 가이드

> 이 문서는 `D:\hankuk\AGENTS.md`를 쉽게 이해하고 실제 작업에 바로 적용하기 위한 설명서입니다.
> 앞으로 새 기능을 만들거나, 기존 서비스를 경찰/소방/지점 단위로 확장할 때 이 문서를 먼저 보면 됩니다.

---

## 1. 가장 중요한 한 줄

앞으로 Hankuk은 이렇게 갑니다.

- 서비스가 다르면 새 앱을 만든다.
- 같은 서비스 안의 경찰/소방/지점/캠퍼스 차이는 기존 앱 안에서 runtime tenant로 처리한다.

쉽게 말하면:

- `study-hall`, `score-predict`, `interview-pass`는 서로 다른 앱
- 그 앱 안의 `police`, `fire`, `gangnam`, `nowon` 같은 것은 tenant/division/workspace

---

## 2. 왜 이 방식이 맞는가

당신이 원하는 방향은 보통 아래를 동시에 만족해야 합니다.

- 서비스는 계속 늘어난다.
- 같은 사용자가 여러 앱을 이동하며 쓴다.
- 로그인은 가능하면 공유하고 싶다.
- 경찰/소방/지점이 늘어나도 앱을 계속 복제하고 싶지는 않다.

그래서 최적 구조는 다음입니다.

### 서비스 축

서비스가 다르면 앱을 나눕니다.

예:

- `study-hall`
- `score-predict`
- `interview-pass`

### 운영 축

같은 서비스 안의 운영 단위는 앱 안에서 나눕니다.

예:

- `study-hall` 안의 `police`, `fire`, `gangnam`
- `interview-pass` 안의 `police`, `fire`

---

## 3. `study-hall`이 기준점인 이유

`study-hall`은 이미 `[division]` 구조를 가지고 있습니다.

즉:

- 라우트가 division 기준으로 나뉘고
- 미들웨어와 권한 검증도 division 기준으로 처리하고
- 같은 앱이 여러 운영 단위를 동시에 다룰 수 있는 방향입니다

그래서 앞으로의 기준은:

- `study-hall`처럼 request 기준으로 tenant를 처리한다
- `NEXT_PUBLIC_TENANT_TYPE`처럼 build 시점에 경찰/소방을 고정하지 않는다

입니다.

---

## 4. 새 앱을 언제 만들까

아래에 많이 해당하면 새 앱이 맞습니다.

- URL이 완전히 분리돼야 한다.
- 화면 구조와 업무 흐름이 많이 다르다.
- 배포 주기가 다르다.
- 운영 책임자가 다르다.
- 나중에 완전히 다른 서비스로 갈 가능성이 크다.

예:

- 자습관 운영 시스템
- 합격예측
- 면접 수강증

이건 서로 새 앱으로 두는 게 맞습니다.

---

## 5. 새 tenant/division은 언제 기존 앱 안에 넣을까

아래에 가까우면 기존 앱 안에서 처리하는 것이 맞습니다.

- 같은 서비스다.
- 같은 코드베이스와 같은 배포 안에서 돌아도 된다.
- 다른 점은 브랜드, 문구, 권한, 지점 설정, 일부 규칙 차이 수준이다.
- 데이터는 tenant/division 필터로 충분히 분리할 수 있다.

예:

- `study-hall` 안에 경찰반 / 소방반 / 강남점 / 노원점
- `interview-pass` 안에 경찰 / 소방

이 경우 새 앱을 만들지 않습니다.

## 6. 지금 상태에서 새 서비스를 추가하는 방법

현재 `D:\hankuk`는 아직 모노레포 전입니다.

즉 지금은:

- 새 서비스 앱을 루트에 바로 만듭니다.
- 각 앱은 npm 기준으로 관리합니다.
- 각 앱은 자기 `.env.local`을 가집니다.
- 지금 당장은 새 앱을 Vercel에 바로 연결하지 않습니다.

예:

```text

  score-predict\
  study-hall\
  interview-pass\
  fire-admin\
```

## 7. 지금 상태에서 기존 앱에 tenant/division을 추가하는 방법

이건 새 앱을 만드는 일이 아닙니다.

예를 들어:

- `study-hall`에 `gangnam` division 추가
- `interview-pass`에 `fire` tenant 추가

이럴 때는 아래를 합니다.

1. 기존 앱에서 runtime 식별자 이름을 정합니다.
   - `division`
   - `tenant`
   - `workspace`
2. path 또는 request 기준으로 분기합니다.
   - 예: `/{division}/...`
3. auth/session에 tenant 문맥이 들어가도록 만듭니다.
4. 모든 DB 쿼리에 tenant/division 필터를 넣습니다.
5. 같은 앱 인스턴스가 여러 tenant를 동시에 처리할 수 있어야 합니다.

---

## 8. DB와 Supabase는 어떻게 생각해야 하나

최종 기준은 다음입니다.

- Supabase 프로젝트는 하나만 쓴다.
- `public` 스키마에는 공통 사용자/인증/권한을 둔다.
- 앱별 데이터는 앱별 스키마에 둔다.
- 같은 앱 안의 경찰/소방/지점은 그 앱 스키마 안에서 `division_id`, `workspace_id`, `tenant_slug` 같은 값으로 구분한다.

즉:

- 새 앱 추가 = 새 스키마 추가 가능
- 같은 앱의 새 tenant 추가 = 새 스키마가 아니라 새 row / 설정 / slug 추가

---

## 9. 도메인은 어떻게 생각해야 하나

최종 기준은 앱별 서브도메인입니다.

예:

- `study.hankukpol.co.kr`
- `score.hankukpol.co.kr`
- `interview.hankukpol.co.kr`

그리고 tenant는 앱 안에서 처리합니다.

예:

- `study.hankukpol.co.kr/police/...`
- `study.hankukpol.co.kr/fire/...`
- `study.hankukpol.co.kr/gangnam/...`

즉:

- 새 서비스 = 새 앱 서브도메인
- 같은 서비스의 새 tenant = 기존 앱 안의 path/slug

---

## 10. build-time tenant 방식은 어떻게 볼까

현재 일부 앱에는 `NEXT_PUBLIC_TENANT_TYPE` 방식이 있습니다.

이건 당장은 남아 있을 수 있지만, 최종 구조로는 권장하지 않습니다.

이유:

- 앱 하나가 한 tenant만 처리하게 된다
- 경찰/소방을 동시에 검증하기 어렵다
- 새 tenant가 늘어날수록 빌드와 배포가 늘어난다

그래서 최종 목표는:

- build-time tenant 고정 -> runtime tenant 처리

입니다.

## 11. 실제로 어떻게 판단하면 되나

### 새 앱으로 간다

- 완전히 다른 제품/서비스다
- 배포와 운영을 따로 가져가야 한다

### 기존 앱 안 tenant로 간다

- 같은 서비스다
- 차이는 운영 단위, 브랜드, 지점, 일부 규칙 차이다

이 두 질문만 기억하면 대부분 맞게 판단할 수 있습니다.

## 13. 흔한 실수

### 실수 1. 경찰/소방이 다르다고 바로 새 앱 만들기

같은 서비스라면 대부분 tenant/division 문제입니다.

### 실수 2. 새 tenant가 생길 때마다 새 Supabase 프로젝트 만들기

지금 최종 전략과 충돌합니다.

### 실수 3. `NEXT_PUBLIC_TENANT_TYPE` 하나로 최종 구조를 굳히기

이건 과도기 방식이지 최종 방식이 아닙니다.

### 실수 4. 같은 서비스인데 Vercel 프로젝트를 tenant마다 따로 만들기

장기적으로 운영 비용과 복잡도만 늘어납니다.

## 14. 최종 요약

앞으로는 이렇게 기억하면 됩니다.

1. 서비스가 다르면 새 앱
2. 같은 서비스의 경찰/소방/지점 차이는 기존 앱 안 tenant
3. 앱별 서브도메인
4. tenant는 앱 안의 path/slug
5. Supabase는 하나, 앱별 스키마 분리
6. 같은 앱 안 tenant는 row/config/filter로 분리
7. `study-hall` 방식이 기준점

## 15. 통합 포털 연동 (새 앱 추가 시 필수)

2026-04부터 통합 관리자 포털(`apps/portal`)이 운영됩니다.
새 앱을 추가하면 아래 단계를 반드시 수행해야 포털에서 해당 앱으로 자동 로그인이 가능합니다.

### 포털 연동 아키텍처

```
[포털 앱]  ── 로그인 ──>  Supabase Auth (이메일/비밀번호)
    │
    ├─ 앱 카드 클릭
    │   ├─ 일회용 launch token 발급 (60초 유효, SHA256 해시 저장)
    │   └─ 대상 앱의 /api/auth/portal-bridge 로 POST 전송
    │
    └─ 대상 앱 bridge
        ├─ consume_portal_launch_token RPC로 토큰 소비 (1회용)
        ├─ 앱 자체 세션 쿠키 발급 (JWT, NextAuth, Supabase Auth 등)
        └─ 관리자 페이지로 리다이렉트
```

- 토큰은 **1회용**이며 60초 후 만료
- 각 앱의 **기존 로그인은 그대로 유지** (폴백)
- 브리지는 **추가 진입 경로**일 뿐, 기존 인증을 대체하지 않음

### 15-1. `@hankuk/config`에 앱 등록

**파일**: `packages/config/src/index.js`

1. `HANKUK_APP_KEYS`에 앱 키 추가:

```javascript
export const HANKUK_APP_KEYS = Object.freeze({
  // ... 기존 키
  MY_NEW_APP: "my-new-app",    // 추가
});
```

2. `HANKUK_SERVICE_CONFIG`에 설정 추가 (portalLaunch 포함):

```javascript
[HANKUK_APP_KEYS.MY_NEW_APP]: Object.freeze({
  appKey: HANKUK_APP_KEYS.MY_NEW_APP,
  displayName: "My New App",
  schemaName: "my_new_app",
  domainAlias: "myapp",
  rootDirectory: "apps/my-new-app",
  productionUrl: "https://my-new-app.vercel.app",
  portalLaunch: Object.freeze({
    superAdminPath: "/super-admin",           // 슈퍼관리자 진입 경로 (선택)
    adminPath: "/{division}/dashboard",       // 관리자 진입 경로
    assistantPath: "/{division}/assistant",   // 조교 진입 경로 (선택)
    staffPath: "/{division}/scan",            // 직원 진입 경로 (선택)
    requiresDivision: true,                   // division 필요 여부
  }),
}),
```

- `{division}`은 포털에서 실제 slug(police, fire 등)로 치환됨
- 지원하지 않는 역할의 경로는 생략하면 해당 역할 카드가 표시되지 않음

3. `packages/config/src/index.d.ts` 타입 정의도 함께 업데이트

### 15-4. 브리지 엔드포인트 구현

**파일 생성**: `apps/my-new-app/src/app/api/auth/portal-bridge/route.ts`

아래 템플릿을 앱의 인증 방식에 맞게 수정:

```typescript
import { createClient } from '@supabase/supabase-js'
import { NextRequest, NextResponse } from 'next/server'

const APP_KEY = 'my-new-app'  // HANKUK_APP_KEYS 값과 동일

type ConsumedPortalLaunch = {
  user_id: string
  division_slug: string | null
  target_path: string
  target_role: 'super_admin' | 'admin' | 'assistant' | 'staff'
}

function createRootServiceClient() {
  const url = process.env.NEXT_PUBLIC_SUPABASE_URL
  const key = process.env.SUPABASE_SERVICE_ROLE_KEY
  if (!url || !key) {
    throw new Error('Supabase environment variables are not configured.')
  }
  return createClient(url, key, {
    auth: { persistSession: false, autoRefreshToken: false },
  })
}

async function consumePortalLaunchToken(token: string) {
  const root = createRootServiceClient()
  const { data, error } = await root.rpc('consume_portal_launch_token', {
    p_plain_token: token,
    p_app_key: APP_KEY,
  })
  if (error) {
    throw new Error(`Failed to consume portal launch token: ${error.message}`)
  }
  const row = Array.isArray(data) ? data[0] : null
  return (row as ConsumedPortalLaunch | null) ?? null
}

function normalizeTargetPath(targetPath: string | null | undefined, fallback: string) {
  if (!targetPath || !targetPath.startsWith('/') || targetPath.startsWith('//')) {
    return fallback
  }
  return targetPath
}

export async function POST(req: NextRequest) {
  try {
    const formData = await req.formData().catch(() => null)
    const launchToken = String(formData?.get('launchToken') || '').trim()
    if (!launchToken) {
      return NextResponse.json({ error: '포털 실행 토큰이 필요합니다.' }, { status: 400 })
    }

    const consumed = await consumePortalLaunchToken(launchToken)
    if (!consumed) {
      return NextResponse.json({ error: '실행 토큰이 유효하지 않거나 만료되었습니다.' }, { status: 401 })
    }

    // ======= 여기부터 앱별 커스텀 로직 =======

    // 1. consumed.target_role 검증 (지원하는 역할만 허용)
    // 2. consumed.user_id로 앱 내부 사용자/관리자 조회
    // 3. 앱 네이티브 세션 쿠키 발급 (JWT, NextAuth, Supabase Auth 등)
    // 4. 리다이렉트

    const fallbackPath = '/dashboard'
    const destination = normalizeTargetPath(consumed.target_path, fallbackPath)
    const response = NextResponse.redirect(new URL(destination, req.nextUrl.origin))

    // response.cookies.set('session_cookie', token, { ... })

    return response

    // ======= 앱별 커스텀 로직 끝 =======

  } catch (error) {
    console.error('Portal bridge error:', error)
    return NextResponse.json(
      { error: '포털 브리지 처리 중 오류가 발생했습니다.' },
      { status: 500 },
    )
  }
}
```

#### 앱 인증 방식별 참고 구현체

| 인증 방식 | 참고 파일 |
|-----------|----------|
| JWT 직접 발급 (PIN 등) | `apps/interview-pass/src/app/api/auth/portal-bridge/route.ts` |
| Prisma + 커스텀 세션 | `apps/study-hall/app/api/auth/portal-bridge/route.ts` |
| Operator 모델 (3단계 역할) | `apps/class-pass/src/app/api/auth/portal-bridge/route.ts` |
| NextAuth JWT | 직접 구현 필요 — `NEXTAUTH_SECRET`으로 JWT 서명 |
| Supabase Auth (SSR) | `supabase.auth.admin.generateLink()` 등으로 세션 생성 |

### 15-5. Vercel 환경변수

새 앱의 Vercel 프로젝트에 아래 환경변수가 설정되어 있어야 브리지가 동작합니다:

| 변수 | 값 | 용도 |
|------|---|------|
| `NEXT_PUBLIC_SUPABASE_URL` | `https://pbonwjwbtqyrfrxqdwlu.supabase.co` | 통합 Supabase URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | (통합 프로젝트 anon key) | Supabase 공개 키 |
| `SUPABASE_SERVICE_ROLE_KEY` | (통합 프로젝트 service role key) | RPC 호출용 |

> 모든 앱은 **동일한 Supabase 프로젝트**를 공유합니다.
> 새 Supabase 프로젝트를 만들지 않습니다.
