# 자동 합격컷 시스템 (Auto Pass-Cut)

> 관련 파일:
> - `src/app/admin/site/auto-pass-cut/page.tsx` — 관리자 설정 UI
> - `src/lib/pass-cut-auto-release.ts` — 핵심 평가·발표 로직 (895줄)
> - `src/lib/pass-cut-release.service.ts` — DB 저장 서비스
> - `src/app/api/admin/pass-cut-release/route.ts` — 관리자 수동 발표 API
> - `src/app/api/internal/pass-cut-auto-release/route.ts` — CRON/트래픽 자동 발표 엔드포인트

---

## 1. 개요

관리자가 직접 합격컷을 입력하지 않아도, 참여 데이터가 일정 조건을 충족하면 **자동으로 합격컷을 계산·발표**하는 기능입니다.

- 발표 횟수: 최대 **1~4차**
- 발표 시 자동으로 공지사항도 함께 생성됨

---

## 2. 트리거 방식 (동작 모드)

| 모드 | 설명 |
|------|------|
| **HYBRID** | 사용자 제출 시 + 외부 CRON 스케줄러 둘 다 가능 (기본값) |
| TRAFFIC_ONLY | 사용자가 OMR 제출할 때만 체크 |
| CRON_ONLY | 외부 CRON 스케줄러에서만 체크 |

**체크 주기** (`site.autoPassCutCheckIntervalSec`, 기본 300초):
- 아무리 자주 제출해도 마지막 체크 이후 N초가 지나야 재실행됨
- 메모리의 `lastTrafficCheckByExamId` Map으로 중복 실행 방지

---

## 3. 발표 조건 판단 로직

각 **지역 × 직렬 × 성별** 조합마다 `READY` 여부를 판정하고,
전체 지역 중 READY 비율이 차수별 기준을 넘으면 자동 발표됩니다.

### 3.1 READY 판정 기준

```
READY 조건:
  1. 참여율(Coverage Rate) ≥ 임계치
     = 실제 참여자 수 / (모집인원 × 합격배수) × 100

  2. 안정도(Stability Score) ≥ 임계치
     = 100 - 합격컷변동페널티 - 신규유입페널티 - 동점자페널티

  두 조건을 모두 충족해야 READY
```

### 3.2 READY가 안 되는 상태 목록

| 상태 | 의미 |
|------|------|
| `COLLECTING_MISSING_APPLICANT_COUNT` | 응시인원 미입력 (관리자가 응시인원을 입력하지 않음) |
| `COLLECTING_INSUFFICIENT_SAMPLE` | 표본 부족 (최소 10명 미만) |
| `COLLECTING_LOW_PARTICIPATION` | 참여율 부족 (임계치 미달) |
| `COLLECTING_UNSTABLE` | 안정도 부족 (합격컷이 아직 흔들리는 중) |
| `READY` | 발표 가능 |

### 3.3 안정도 계산 상세

```
안정도 = 100 - 합격컷변동페널티 - 신규유입페널티 - 동점자페널티

합격컷변동페널티 = min(40, |현재 1배수 컷 - 1시간 전 1배수 컷| × 20)
신규유입페널티   = min(30, 최근1시간 신규유입률(%) × 1.5)
동점자페널티     = min(30, 1배수 컷 동점자 비율(%) × 1.2)
```

## 4. 임계치 프로파일

### 4.1 참여자수 임계치 프로파일 (`thresholdProfile`)

| 프로파일 | 1차 참여율/안정도 | 2차 | 3차 | 4차 | 최소표본 |
|---------|----------------|-----|-----|-----|---------|
| **BALANCED** | 30% / 45 | 50% / 55 | 70% / 65 | 90% / 75 | 10명 |
| CONSERVATIVE | 40% / 55 | 60% / 65 | 80% / 75 | 95% / 85 | 15명 |
| AGGRESSIVE | 20% / 35 | 35% / 45 | 50% / 55 | 70% / 65 | 8명 |

### 4.2 준비비율 임계치 프로파일 (`readyRatioProfile`)

N차 발표 시 몇 %의 지역이 READY 상태여야 하는지 결정

| 프로파일 | 1차 | 2차 | 3차 | 4차 |
|---------|-----|-----|-----|-----|
| **BALANCED** | 25% | 45% | 65% | 85% |
| CONSERVATIVE | 35% | 55% | 75% | 95% |
| AGGRESSIVE | 15% | 30% | 50% | 70% |

> **예시**: BALANCED + BALANCED 설정 → 1차 발표는 전체 지역 중 25% 이상이 READY이면 발표

## 5. 전체 실행 흐름

```
runAutoPassCutRelease(trigger)
  │
  ├─ 1. 자동 발표 활성화 여부 확인 (autoPassCutEnabled)
  ├─ 2. 모드 체크: HYBRID/TRAFFIC_ONLY/CRON_ONLY vs 현재 트리거
  ├─ 3. 활성 시험(isActive) 조회
  ├─ 4. (TRAFFIC 트리거) 체크 주기 간격 확인
  ├─ 5. 다음 발표 번호 확인 (1~4차, 이미 완료된 것 제외)
  ├─ 6. evaluateRows() → 지역·직렬·성별별 READY 여부 판정
  ├─ 7. readyRegionRatio 계산 → 차수별 임계치와 비교
  └─ 8. 조건 충족 시 createPassCutRelease() → DB 저장 + 공지 생성
```

### 5.1 reason 반환값 (왜 발표 안 됐는지 확인 가능)

| reason | 의미 |
|--------|------|
| `RELEASE_CREATED` | ✅ 발표 생성 완료 |
| `AUTO_DISABLED` | 자동 발표 비활성화 상태 |
| `MODE_BLOCKED` | 현재 트리거가 설정된 모드와 불일치 |
| `INTERVAL_THROTTLED` | 체크 주기 미경과 |
| `NO_ACTIVE_EXAM` | 활성 시험 없음 |
| `THRESHOLD_NOT_REACHED` | 준비 지역 비율 미달 |
| `ALL_RELEASES_COMPLETED` | 4차 모두 발표 완료 |
| `DUPLICATED` | 해당 차수 이미 발표됨 |

## 6. 발표 시 저장 데이터

### PassCutRelease 테이블
```
id, examId, releaseNumber(1~4), releasedAt, participantCount, createdBy
source: "AUTO" | "ADMIN"
```

### PassCutSnapshot 테이블 (지역 × 직렬 × 성별별 1행)
```
regionId, examType, gender
status, statusReason
participantCount, recruitCount, applicantCount
coverageRate, stabilityScore
oneMultipleCutScore   ← 1배수 합격선 (핵심)
sureMinScore          ← 확실권 최저점
likelyMinScore        ← 유력권 최저점
possibleMinScore      ← 가능권 최저점
averageScore
```

## 7. 공지 자동 생성

발표 생성 시 공지사항이 자동으로 생성됩니다.

```
제목: "2차 합격컷 자동 발표 안내"
우선순위: 110 (관리자 수동 발표 100보다 높음)
isActive: true

내용 예시:
  2차 합격컷이 자동 발표되었습니다.
  충족 지역: 15/18 (83.3%)
  미집계 지역: 3건

  [지역·직렬별 상태]
  - 서울-공채(남): 집계완료 (1배수컷 150.50점) ...
  - 대구-공채(남): 미집계 - 참여율 부족 45.2% < 50% ...
```

## 8. CRON 설정 방법 (HYBRID / CRON_ONLY 모드 시)

외부 CRON 서비스(Vercel Cron, GitHub Actions 등)에서 아래 엔드포인트를 호출합니다.

```bash
# 5분마다 호출 예시
POST https://{domain}/api/internal/pass-cut-auto-release
Authorization: Bearer {AUTO_PASSCUT_CRON_SECRET}
Content-Type: application/json

{ "force": false }
```

### 필요한 환경 변수

```env
AUTO_PASSCUT_CRON_SECRET=your_secret_here   # CRON 인증 토큰 (필수)
AUTO_PASSCUT_ADMIN_USER_ID=1                # 자동 발표 생성자 ID (선택, 없으면 첫 번째 ADMIN 사용)
```

### Vercel Cron 설정 예시 (`vercel.json`)

```json
{
  "crons": [
    {
      "path": "/api/internal/pass-cut-auto-release",
      "schedule": "*/5 * * * *"
    }
  ]
}
```

## 9. 관리자 수동 발표

자동 발표 조건이 충족되지 않아도 관리자가 강제 발표할 수 있습니다.

- **경로**: `/admin/pass-cut` 페이지
- **API**: `POST /api/admin/pass-cut-release`
- 수동 발표 시 `source: "ADMIN"` 으로 저장됨

---

## 10. 현재 설정값 (기본값)

| 설정 키 | 기본값 | 의미 |
|--------|--------|------|
| `site.autoPassCutEnabled` | `false` | 자동 발표 비활성화 |
| `site.autoPassCutMode` | `HYBRID` | 켜면 트래픽+CRON 모두 동작 |
| `site.autoPassCutCheckIntervalSec` | `300` | 5분마다 조건 체크 |
| `site.autoPassCutThresholdProfile` | `BALANCED` | 참여율/안정도 기준 |
| `site.autoPassCutReadyRatioProfile` | `BALANCED` | 준비 지역 비율 기준 |

## 11. 운영 시나리오

### 정답 발표 당일 권장 절차

1. **정답 입력** (`/admin/answers`) → 전체 제출 자동 재채점
2. **자동 발표 활성화** 체크 → 5분 간격으로 조건 자동 체크
3. 참여자 증가 → 조건 충족 시 1차, 2차... 자동 발표
4. 4차 발표 완료 또는 시험 마감 후 자동 발표 비활성화

### 빠른 발표가 필요할 때
- `AGGRESSIVE` 프로파일로 변경 → 낮은 기준으로 더 빨리 발표
- 또는 관리자가 수동 강제 발표

### 신중한 발표가 필요할 때
- `CONSERVATIVE` 프로파일로 변경 → 높은 기준으로 충분히 안정된 후 발표
