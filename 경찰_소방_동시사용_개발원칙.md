# 경찰_소방_동시사용_개발원칙

> 최우선 기준: `D:\hankuk\AGENTS.md`
> 목적: 경찰/소방/지점/캠퍼스 같은 운영 단위를 같은 서비스 안에서 어떻게 함께 운영할지 빠르게 판단하기 위한 보조 문서

---

## 1. 한 줄 기준

- 서비스가 다르면 새 앱이다.
- 같은 서비스 안의 경찰/소방/지점 차이는 같은 앱 안의 runtime tenant다.

즉:

- `academy-ops`, `study-hall`, `score-predict`, `interview-pass`는 서로 다른 서비스 앱
- 각 앱 안의 `police`, `fire`, `gangnam`, `nowon` 같은 것은 tenant/division/workspace

---

## 2. 왜 이렇게 가야 하나

- 배포와 운영은 서비스 단위로 분리하고 싶다.
- 같은 사용자가 여러 서비스를 오가며 써야 한다.
- 경찰/소방 직렬이 늘어나도 앱과 프로젝트를 계속 복제하고 싶지 않다.
- 공통 인증과 사용자 정보를 최대한 공유하고 싶다.

그래서 Hankuk에서는:

- 앱 수는 서비스 경계만큼만 늘리고
- 같은 서비스 안의 운영 차이는 runtime scope로 흡수한다.

---

## 3. runtime tenant 처리 기준

권장 순서:

1. path slug
2. runtime hostname lookup
3. 기타 request-scoped lookup

예:

- `/police/...`
- `/fire/...`
- `/gangnam/...`
- `academy.hankukpol.co.kr`에서 hostname 또는 `academyId`로 활성 운영 단위 결정

핵심:

- 하나의 앱 인스턴스가 여러 tenant를 동시에 처리해야 한다.
- tenant마다 별도 빌드와 별도 배포를 만들지 않는다.
- 인증 문맥과 DB 조회 범위에 tenant 정보가 항상 포함되어야 한다.

---

## 4. 경찰/소방 직렬 차이로 분리해도 되는 것

같은 앱 안에서 tenant 설정이나 데이터로 분리할 수 있는 것:

- 브랜드명, 로고, 색상
- 기본 문구, 공지 노출 범위
- 권한 범위와 운영자 소속
- 시험 과목 카탈로그
- 수납 정책, 할인 정책, 시설 설정
- 지점/캠퍼스/직렬별 통계 집계 범위

이런 차이는 `config`, `settings`, `academyId`, `division`, `tenant_slug` 같은 값으로 처리한다.

---

## 5. 경찰/소방 직렬 차이로 분리하면 안 되는 것

아래 이유만으로는 새 앱을 만들지 않는다.

- 경찰용 UI와 소방용 UI 문구가 조금 다르다
- 직렬별 과목 구성이 다르다
- 직렬별 공지나 권한이 다르다
- 지점별 설정과 운영 방식이 다르다
- 같은 서비스인데 대상 집단만 다르다

금지:

- 경찰/소방 차이만으로 새 앱 생성
- 경찰/소방 차이만으로 새 Vercel 프로젝트 생성
- 경찰/소방 차이만으로 새 Supabase 프로젝트 생성

---

## 6. 앱별 해석

### `study-hall`

- 기준 패턴이다.
- `[division]` 같은 request-scoped 라우팅이 가장 직관적이다.

### `score-predict`

- 현재 build-time tenant 흔적이 남아 있다.
- 장기적으로는 runtime tenant 구조로 이동해야 한다.
- 계산 규칙 차이는 내부 모듈 분리로 해결하고 앱 자체는 하나로 유지한다.

### `interview-pass`

- 현재 build-time tenant 흔적이 남아 있다.
- 경찰/소방 차이는 config, 라벨, 권한, request 문맥으로 옮기는 것이 목표다.

### `academy-ops`

- 학원 통합 운영 메인 프로그램이라는 하나의 서비스 경계다.
- 경찰/소방/지점/캠퍼스 차이는 이 앱 안에서 처리한다.
- 현재처럼 `academyId`, `hostname`, 지점별 설정을 이용한 row-based 멀티테넌시가 맞는 방향이다.
- 필요하면 path slug를 추가하되, 별도 sibling app으로 분리하지 않는다.

---

## 7. DB 원칙

- Supabase 프로젝트는 하나를 쓴다.
- 공통 인증과 사용자 정보는 `public`에 둔다.
- 앱별 데이터는 앱별 schema로 둔다.
- 같은 앱 안의 경찰/소방/지점 차이는 row/config/filter로 분리한다.

예:

- `academy_ops` schema 안에서 `academyId`나 `tenant_slug`로 구분
- `study_hall` schema 안에서 `division_id`로 구분
- `interview` schema 안에서 `tenant_slug`로 구분

정리:

- 새 앱 추가 = 새 schema 추가 가능
- 같은 앱의 새 직렬/지점 추가 = 새 schema가 아니라 새 row / 새 설정 / 새 slug

---

## 8. 구현 체크리스트

- 라우팅이 tenant 문맥을 만들고 있는가
- 인증/session이 tenant를 들고 있는가
- 모든 DB 조회와 생성에 tenant 경계가 들어가는가
- tenant별 문구/브랜딩/설정이 config나 데이터로 분리되는가
- 하나의 앱 인스턴스가 경찰과 소방을 동시에 처리할 수 있는가

---

## 9. 한 줄 결론

Hankuk의 원칙은 단순합니다.

- 앱은 서비스별로 분리한다.
- 경찰/소방/지점 차이는 같은 앱 안에서 runtime tenant로 처리한다.
- `academy-ops`도 예외가 아니라 같은 원칙을 따른다.
